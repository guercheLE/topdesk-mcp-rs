# MCP prompts: guided TOPdesk management workflows

## Context

`topdesk-mcp-rs` currently exposes exactly 3 MCP tools — `search`, `get`, `call` — backed by an embedded, per-API-version catalog of TOPdesk API operations ([src/core/mcp_server.rs](../src/core/mcp_server.rs)). This is 19 separate catalogs, one per TOPdesk product module/version (`general-1.2.0` default, plus `incident-4.2.6`, `change-1.4.0`, `assets-1.91.1`, `knowledge-base-before-explorer-migration`, `knowledge-base-after-explorer-migration`, `knowledgebase-graphql-1.0.0`, `reservations-2.0.0`, `services-1.3.7`, `operations-management-1.10.0`, `visitors-2.0.0`, `supporting-files-2.7.11`, `access-roles-saas`, `access-roles-va-release-1-2026`, `lookandfeel-1.0.0`, `task-notifications-1.0.0`, `settings-1.1.0`, `custom-action-support-saas`, `custom-action-support-va-release-1-2023-or-newer` — full table in [docs/SCHEMA_VERSIONS.md](SCHEMA_VERSIONS.md)), each with its own embedded SQLite operation-catalog (`mcp_store_v*.db.zst`) and JSON-schema bundle (`generated_schemas_v*.json.zst`).

Unlike a project where "version" means a revision of the *same* API, here each of the 19 labels is a genuinely different TOPdesk *product module* — Incident Management, Change Management, Asset Management, etc. **Only one catalog is active per running server instance**, selected by the `api_version` config value at startup and not switchable mid-session. This is the load-bearing constraint the whole prompts design has to respect: a guided workflow written for Change Management is silently useless if the server actually started with `general-1.2.0` (or any other non-`change-1.4.0`) catalog active — `search` simply won't find its operations, and there is no way for the calling LLM to fix that itself (it requires a config change + process restart).

This flat 3-tool surface is powerful but leaves all workflow sequencing — "classify the change, get planning/backout documented, route through CAB approval before touching an execute step, log outcome, run a post-implementation review" — entirely up to whichever LLM is driving the client, re-derived from scratch every session, with no guardrails against skipping an approval gate or silently reordering an emergency-change fork.

The goal is to add an MCP **prompts** capability: a master "menu" prompt plus one prompt per logical TOPdesk domain (incident lifecycle, change management, asset management, knowledge base, reservations, reference-data lookup, visitor registration, operations-management tasks, access-roles assignment). Each prompt returns instructional prose that guides the calling LLM through a domain's task step by step — asking for missing parameters, gating progression at points where TOPdesk's real semantics require a decision (e.g. an approval a human actually has to have made, not one this tool should self-grant), calling out where steps are independent and can run in parallel, and always describing operations by capability ("search for how to create a request for change") rather than by a specific `operationId` or assumed response shape, since the catalog is semantically searched rather than a fixed script.

This repo is generated output from an external `mcpify` generator (every existing `.rs` file is headed `// General API MCP server — generated by mcpify. Do not hand-edit.`; `mcpify.yaml` drives regeneration). Recent git history (`fix(rust):`, `test(rust):`, `perf(ci):` commits layered on top of the generated baseline) shows hand-editing this repo directly is already this project's normal practice, not a special exception — so this feature follows that same precedent rather than requiring generator changes.

The mechanism is already available for free: this crate's `rmcp = "2"` dependency resolves to **rmcp 2.1.0**, which ships a first-class prompts API (`Prompt`, `PromptArgument`, `PromptMessage`, `GetPromptResult`, `#[prompt_router]`, `#[prompt]`, `#[prompt_handler]`) that mirrors the `#[tool_router]`/`#[tool]`/`#[tool_handler]` pattern [src/core/mcp_server.rs](../src/core/mcp_server.rs) already uses for `search`/`get`/`call` almost exactly. Verified directly against the vendored crate sources (`rmcp-2.1.0` and `rmcp-macros-2.1.0`, including the macro implementation itself, not just its tests): `#[prompt_router]` appends a `prompt_router() -> PromptRouter<Self>` associated fn to whatever inherent `impl <Self>` block it decorates (it doesn't need to be the same block as `#[tool_router]`), `PromptRouter<S>` is `Clone`, and `#[tool_handler]`/`#[prompt_handler]` are designed to stack on the same `impl ServerHandler` block — each only pushes its own disjoint set of trait methods (`call_tool`/`list_tools` vs. `get_prompt`/`list_prompts`) and both explicitly skip auto-generating `get_info()` when a hand-written one (like this server's) already exists, so `.enable_prompts()` has to be added to the existing builder call by hand — which is exactly the correct approach here, not a workaround. `rmcp`'s default features already include `macros` (the feature gating `#[prompt]`/`#[prompt_router]`/`#[prompt_handler]`, same as it gates the existing `#[tool]` macros), so no `Cargo.toml` change is needed.

The constructor `McpifyServer::new(api_version, config, auth_manager)` does **not** need a new parameter — it can build `prompt_router: Self::prompt_router()` internally exactly like it already builds `tool_router: Self::tool_router()`. There is no struct-literal construction of `McpifyServer` anywhere in the codebase, only `::new(...)` calls (`src/main.rs:147`, `src/main.rs:161`, `src/core/mcp_server.rs:249` test helper, `src/http/server.rs:279` test helper) — none of these four call sites need to change.

## Approach

### File layout

Prompt code is kept entirely separate from tool code: all prompt logic lives in a new `src/prompts/` module, distinct from `src/tools/` (which holds `search`/`get`/`call`'s business logic — [search_tool.rs](../src/tools/search_tool.rs), [get_tool.rs](../src/tools/get_tool.rs), [call_tool.rs](../src/tools/call_tool.rs)). The `#[prompt_router]`-decorated `impl McpifyServer` block goes in `src/prompts/router.rs`, never in the same file as the existing `#[tool_router]`-decorated block at [mcp_server.rs:76](../src/core/mcp_server.rs). [src/core/mcp_server.rs](../src/core/mcp_server.rs) itself is touched only for the minimal wiring a single `ServerHandler`/struct necessarily requires — new struct field, a small `pub(crate) fn api_version(&self)` accessor, the stacked handler macro, and the `.enable_prompts()` capability flag — no prompt method bodies or prompt-specific logic land there.

New module `src/prompts/`, declared from `src/lib.rs` next to the existing `pub mod tools;` (line 16):

```
src/prompts/

                                       // delegation_guidance() (+ their own #[cfg(test)] unit tests)
  router.rs                           // second `impl McpifyServer` block, #[prompt_router]-decorated
  content/
    menu.md                           // written LAST, once every prompt name is final
    incident_lifecycle.md
    change_management_lifecycle.md    // worked example, written FIRST — see Sequencing
    asset_management_basics.md
    knowledge_base_lifecycle.md
    reservations_booking.md
    reference_data_lookup.md
    visitor_registration.md
    operations_management_tasks.md
    access_roles_assignment.md
```

Instructional prose lives in `.md` files pulled in via `include_str!`, not inline Rust string literals — this follows the pattern the crate already uses for large embedded assets ([data/store.rs](../src/data/store.rs) `include_bytes!`s each version's `.db.zst`; [validation/validator.rs](../src/validation/validator.rs) does the same for schema JSON), adapted from `include_bytes!` (binary) to `include_str!` (text) since this is the first text-content embedding in the crate. Anything that varies per-invocation (the active `api_version`, whichever context the caller already supplied) is rendered separately in Rust and prepended/appended to the static markdown body — no template-substitution engine needed.

New hand-authored files do **not** carry the `// General API MCP server — generated by mcpify. Do not hand-edit.` header every existing file has (that claim would be false — mcpify's generator has no concept of prompts). Instead each new file opens with a one-line note that it's hand-authored and won't be touched by a `mcpify` regeneration.

### `src/prompts/mod.rs` — shared helpers and argument struct

```rust
#[derive(Debug, Deserialize, schemars::JsonSchema)]
pub struct WorkflowArgs {
    /// Anything you already know going in: a ticket/record number, current

```

One shared, generic arg struct reused by all 9 domain prompts (not 9 separate structs) — less code, consistent shape across every prompt, and every domain-specific meaning ("existing change number" vs. "existing incident number") is explained in that prompt's own `.md` body instead of in the argument name.

Three pure-Rust helper functions, each independently unit-testable with no transport/server dependency:

```rust

/// has one `api_version` catalog active per process (config-time, not
/// switchable mid-session — see `core::config_schema::Config::api_version`

/// written for e.g. `change-1.4.0` is silently useless if the operator
/// actually started the server with a different catalog active. Stating

/// discovering the mismatch the hard way, or worse, inventing an
/// operationId when `search` comes back empty.
pub(crate) fn catalog_check_block(active_api_version: &str, expected: &[&str]) -> String { /* ... */ }

/// Renders the "context already provided" header from the optional
/// `WorkflowArgs.context` field — empty string if `None` or blank.
pub(crate) fn render_context_header(context: Option<&str>) -> String { /* ... */ }

/// Static delegation/parallelization guidance appended to every prompt —
/// see "Delegation guidance" below.
pub(crate) fn delegation_guidance() -> &'static str { /* ... */ }
```

`catalog_check_block` behavior: if `active_api_version` is in `expected`, render a short "matches, proceed, but still confirm each operation via `search`/`get`" note; otherwise render a "MISMATCH — do not proceed, tell the user to restart with a different `api_version`, see docs/SCHEMA_VERSIONS.md" note. Unit tests cover both branches plus `render_context_header`'s `None`/blank/populated cases and a sanity check that `delegation_guidance` mentions whole-workflow delegation.

### `src/prompts/router.rs` — one method per prompt

Mirrors the existing `SearchArgs`/`GetArgs`/`CallArgs` + `#[tool(...)]` pattern:

```rust
#[prompt_router]
impl McpifyServer {
    #[prompt(
        name = "topdesk-menu",
        description = "Directory of every guided TOPdesk workflow prompt this server offers, plus which API-version catalog each one needs."
    )]
    async fn topdesk_menu(&self) -> Vec<PromptMessage> {
        let body = format!(
            "Active catalog right now: `{}`\n\n{}\n{}",
            self.api_version(),
            include_str!("content/menu.md"),
            delegation_guidance(),
        );
        vec![PromptMessage::new_text(Role::User, body)]
    }

    #[prompt(
        name = "change-management-lifecycle",
        description = "Guided RFC lifecycle for TOPdesk Change Management: classification, planning, CAB approval routing, implementation, and closure."
    )]
    async fn change_management_lifecycle(
        &self,
        Parameters(args): Parameters<WorkflowArgs>,
    ) -> Vec<PromptMessage> {
        let body = format!(
            "{}{}{}\n{}",
            catalog_check_block(self.api_version(), &["change-1.4.0"]),
            render_context_header(args.context.as_deref()),
            include_str!("content/change_management_lifecycle.md"),
            delegation_guidance(),
        );
        vec![PromptMessage::new_text(Role::User, body)]
    }

    // 7 more domain prompts, identical shape, one expected-catalog list each
}
```

`topdesk-menu` takes no arguments (omits the `Parameters<T>` extractor — the macro emits `arguments: None` automatically); every domain prompt takes `Parameters<WorkflowArgs>`, and `context` is `Option<String>` so rmcp advertises it as `required: false` — appropriate since MCP prompt arguments are conventionally collected up front by a client UI, not well suited to values that only become known partway through a guided flow; pushing "ask if missing" into the instructional prose instead of transport-level required-argument validation is what makes this work uniformly for agentic clients that never populate prompt arguments at all, and interactive ones whose humans do.

### The "one active catalog" rule (applies to every prompt)

This is the TOPdesk-specific analogue of API schema drift: instead of "the same operation might have a different shape depending on version" (not a concern here, since each label is a distinct product module, not a revision of one API), the concern is "the operation might not exist *at all* in whatever single catalog this process loaded." Every domain prompt's body opens with `catalog_check_block`'s rendered output (ground truth from `self.api_version()`, zero tool round-trips) before any step-by-step content, and any step that plausibly needs a different catalog than the one the prompt is nominally about (e.g. change-management's execution checklist living in `operations-management-1.10.0`, or its incident-linkage check needing `incident-4.2.6`) must say so explicitly rather than silently assuming success. Treat this as a hard rule to check for in review, not just a default — a prompt that doesn't open with the catalog check, or that references a cross-catalog operation without flagging it as such, is incomplete.

### Every operation reference is a search phrase, never a hardcoded operationId

Since the embedded catalog is semantically searched and its exact operationIds/schemas aren't a fixed contract the prompts can assume ahead of time, every domain prompt is written as a **playbook over `search`/`get`/`call`**, not a literal API script: each step says "search for an operation like '\<phrase\>', inspect its schema with `get`, then `call` it with the fields it requires" rather than naming a specific `operationId`. This keeps prompt content correct even if the embedded store changes shape later — only wording could go stale, never a broken hardcoded call.

### Delegation and parallelization guidance

`delegation_guidance()` (in `src/prompts/mod.rs`, appended once per prompt by `router.rs` — not duplicated per `.md` file) covers two levels, phrased conditionally since not every MCP client has a sub-task mechanism:

- **Whole-workflow delegation** (primary lever for saving context/tokens): once the catalog check passes, hand the entire prompt to an isolated sub-task if the environment supports one, and have it report back only the outcome (record number, status, anything that needs a human decision) — not every intermediate `search`/`get`/`call` round-trip.
- **Step-level delegation/parallelization**: large `search` sweeps (enumerating every category/operator/branch), and independent bulk `call`s with no ordering dependency (registering several unrelated assets, creating the DLX-equivalent independent resources within one workflow), are candidates to delegate or run concurrently rather than piping every raw row through the main context.

### Worked example: `change-management-lifecycle`

Change Management has 56 endpoints in the real catalog (per `docs/topdesk-api-specs.md`) — the richest workflow domain by a wide margin. `src/prompts/content/change_management_lifecycle.md` (target ~120–220 lines, workflow-specific prose only — the catalog-check/context-header/delegation wrapper is injected separately and doesn't count against this budget) follows this shape:

1. **Classify the RFC** — standard/pre-approved, simple, extensive, or emergency; ask the user or infer from what they've said. Standard skips straight to creation; extensive needs the full risk/impact + CAB path; emergency forks (see 4a).
2. **Create the RFC** — search/get/call to create it with description, category, requester, change type from step 1. Capture and surface the returned change number immediately, since it's how the workflow resumes if interrupted.
3. **Attach implementation and backout plans** — required for extensive changes, recommended even for simple ones; a backout plan with no owner or rollback trigger isn't a real one, don't let the user skip it silently on an extensive change.
4. **Impact & risk assessment** (extensive only) — check via `get` what the schema actually records, don't assume a field name.
   - **4a. Emergency fork** — if the change is to stop active harm, TOPdesk's real-world pattern is implement-first/retrospective-approval; require the user's explicit confirmation before reordering steps 4 and 5, since skipping pre-approval has real audit consequences.
5. **Approval gate — stop here, do not self-approve.** Approvals are typically their own sub-records/tasks, possibly CAB-routed with multiple approvers. Before calling anything approval-shaped: check what an authorization step actually looks like in this schema, ask the user who the real approvers are and whether approval already happened outside this tool, and only call it on the user's explicit instruction that a named approver actually approved it. This tool records human decisions, it doesn't make them.
6. **Schedule the implementation window** — often modeled as an activity/task on the change, not a field on the change record itself.
7. **Build, test, execute** — cross-catalog note: the execution checklist may live in `operations-management-1.10.0`, a separate catalog from `change-1.4.0`; if that catalog isn't active, tell the user rather than guessing at an operationId that isn't loaded.
8. **Log actual start/end and outcome** (success/partial/failed/backed out).
9. **Post-implementation review** — lessons learned, and whether the change caused any linked incidents (another cross-catalog concern — needs `incident-4.2.6`, not `change-1.4.0`).
10. **Close the change.**

This is the concrete illustration of every element the design calls for: catalog-check discipline (steps 7, 9), an explicit classification fork (step 1), an approval gate the tool must not bypass (step 5), and an emergency-path fork requiring explicit human confirmation (step 4a). Every other domain prompt follows the same skeleton at a scope proportional to its own real complexity — numbered steps, explicit gates where TOPdesk's actual semantics require one, agnostic search-language instructions, and any genuinely independent sub-steps called out as parallelizable.
